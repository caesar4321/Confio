# Coinbase Ramp Engineering Plan

Generated by /plan-eng-review on 2026-06-27
Branch: main
Repo: Confio
Status: DRAFT

## Goal

Add Coinbase as the preferred low-cost US ACH ramp provider while keeping Guardarian hosted card/mobile-pay as the fallback for US users who cannot or do not want to use Coinbase ACH.

US users should be able to fund their Confio non-custodial Algorand wallet with USDC through:

- Coinbase hosted Onramp for logged-in Coinbase users and ACH where eligible.
- Guardarian hosted card/mobile-pay for debit card, Apple Pay, and Google Pay where available.

V1 prioritizes onramp. Offramp is designed into the provider foundation but does not block onramp launch.

## Step 0: Scope Challenge

### What Already Exists

- `ramps.models.RampTransaction` already stores provider, direction, status, actor, fiat/crypto amounts, provider IDs, metadata, and links to USDC deposit/withdrawal/conversion records.
- `ramps.models.RampWebhookEvent` already gives idempotent webhook storage. Idempotent means a repeated provider event should be harmless and processed once.
- `ramps.schema.CreateRampOrder` already resolves the active JWT account, validates wallet readiness, gets the Algorand destination address, calls the provider, and creates a `RampTransaction`.
- `ramps.schema.resolve_ramp_availability`, `resolve_ramp_quote`, and `resolve_ramp_order_status` already define the mobile contract.
- `ramps.views.koywe_webhook` already shows the webhook pattern: parse, verify, store event, reconcile provider status, update `RampTransaction`.
- `apps/src/screens/TopUpScreen.tsx` already has the modern native ramp UX.
- `apps/src/screens/LegacyGuardarianTopUpScreen.tsx` is the fallback for non-Koywe countries.

### Minimum Useful Change

Add a provider-aware US onramp choice without rewriting the whole ramp domain:

1. Add `coinbase` to provider choices.
2. Add US payment method definitions:
   - `COINBASE_HOSTED_ACH`: Coinbase login, ACH handled by Coinbase where eligible.
   - `GUARDARIAN_HOSTED_CARD`: Guardarian card/mobile-pay hosted checkout.
3. Add a `CoinbaseClient` and Coinbase sync helpers matching the current Koywe shape.
4. Keep Guardarian as a selectable US method instead of only a legacy fallback.
5. Route `US` TopUp through the modern ramp screen instead of hard-falling into `LegacyGuardarianTopUpScreen`.
6. Store and reconcile Coinbase orders through `RampTransaction`.
7. Add Coinbase webhook endpoint and tests.

Do not build a generic provider registry in V1. The existing code is Koywe-shaped, but a full registry is more refactor than this feature needs.

### Complexity Check

This plan will touch more than 8 files if implemented fully. That is acceptable because the blast radius is naturally split across backend provider, webhook, tests, and mobile routing. The scope reduction is to avoid a full provider abstraction and avoid Coinbase off-ramp UI in V1.

## Recommended Architecture

Use the existing `ramps` domain and add Coinbase as a second first-class provider.

```
US user taps Recargar
        |
        v
TopUpScreen country=US
        |
        v
GET_RAMP_AVAILABILITY
        |
        +--> US methods:
             - COINBASE_HOSTED_ACH (recommended/default)
             - GUARDARIAN_HOSTED_CARD (fallback)
        |
        v
GET_RAMP_QUOTE
        |
        v
CREATE_RAMP_ORDER(provider inferred by method)
        |
        +--> server resolves active Account from JWT
        +--> server reads account.algorand_address
        +--> Coinbase/Guardarian hosted order uses:
             destinationAddress=<Confio Algorand address>
             destinationNetwork=algorand
             asset=USDC
        |
        v
RampTransaction(provider=coinbase|guardarian, status=PENDING)
        |
        +--> hosted response: nextActionUrl
        |
        v
Provider webhook/status poll
        |
        v
RampTransaction COMPLETED/FAILED/AML_REVIEW
        |
        v
existing USDC deposit/conversion flow detects wallet funding
```

## Backend Plan

### 1. Settings

Add settings with sandbox/prod separation:

- `COINBASE_ONRAMP_API_URL`
- `COINBASE_ONRAMP_API_KEY` or CDP key material required by the selected API
- `COINBASE_ONRAMP_WEBHOOK_SECRET`
- `COINBASE_ONRAMP_PROJECT_ID`
- `COINBASE_ONRAMP_DEFAULT_NETWORK=algorand`
- `COINBASE_ONRAMP_DEFAULT_ASSET=USDC`
- `COINBASE_ONRAMP_ENABLED`

Use AWS Secrets Manager for production secrets, matching the Koywe and Guardarian pattern in `config/settings.py`.

### 2. Models

Update provider choices:

- `RampTransaction.PROVIDER_CHOICES += ('coinbase', 'Coinbase')`
- `RampWebhookEvent.PROVIDER_CHOICES += ('coinbase', 'Coinbase')`

Do not add a new Coinbase transaction table in V1. Put provider-specific payloads in `RampTransaction.metadata`.

Suggested metadata keys:

```json
{
  "payment_method_code": "COINBASE_HOSTED_ACH",
  "destination_address": "ALGO...",
  "destination_network": "algorand",
  "asset": "USDC",
  "order_payload_created": {},
  "order_payload_latest": {},
  "checkout_mode": "hosted",
  "coinbase_status": "..."
}
```

### 3. Coinbase Client

Create `ramps/coinbase_client.py`.

Responsibilities:

- Build authenticated Coinbase API requests.
- Fetch buy options for `US + USDC + Algorand` and cache briefly.
- Create hosted onramp session/order.
- Fetch order status.
- Normalize Coinbase errors into Spanish-safe product errors.

Do not import mobile concepts here. Return dataclasses like Koywe:

- `CoinbaseQuoteResult`
- `CoinbaseOrderResult`
- `CoinbaseOrderStatusResult`

### 4. Coinbase Sync

Create `ramps/coinbase_sync.py`.

Responsibilities:

- Verify webhook signatures.
- Extract event ID.
- Extract order/session ID.
- Map provider statuses to Confio statuses:
  - pending/created -> `PENDING`
  - payment processing / crypto pending -> `PROCESSING`
  - completed / delivered -> `COMPLETED`
  - failed/cancelled/refunded -> `FAILED`
  - review/held/compliance review -> `AML_REVIEW`
- Upsert/update `RampTransaction`.

The status mapper must be table-driven and unit-tested because provider strings drift over time.

### 5. GraphQL

Keep the existing GraphQL names:

- `rampAvailability`
- `rampQuote`
- `createRampOrder`
- `rampOrderStatus`

Add provider inference from `payment_method_code`:

```text
COINBASE_HOSTED_ACH -> coinbase
GUARDARIAN_HOSTED_CARD -> guardarian
everything else -> koywe
```

Do not require the mobile client to pass `provider` in V1. The server already owns the payment catalog, so method code is enough and avoids spoofing.

Add US provider behavior:

- US Coinbase onramp does not require `RampUserAddress` in V1 unless Coinbase explicitly requires address data for the hosted flow.
- US onramp still requires a valid active Algorand address and wallet backup readiness.
- Coinbase hosted method returns `nextActionUrl`.
- Guardarian hosted method can reuse the existing Guardarian transaction proxy logic initially, but should write a `RampTransaction(provider=guardarian)` so history and pending-state behavior match Coinbase/Koywe.

### 6. Webhook

Add route:

- `POST /api/coinbase/onramp/webhook/`

Flow:

```
request
  |
  +-- parse JSON
  +-- verify Coinbase signature
  +-- get_or_create RampWebhookEvent(provider=coinbase, event_id=...)
  +-- duplicate? return ok
  +-- find RampTransaction(provider=coinbase, provider_order_id=...)
  +-- missing tx? store event and return ok
  +-- fetch latest status if webhook payload is not enough
  +-- update RampTransaction
```

Never let an unknown order or duplicate webhook return a provider-visible failure. Store it, log it, and return 200.

## Mobile Plan

### 1. Provider-Aware Routing

Replace `isKoyweRoutingEnabledForCountry(countryCode)` in `TopUpScreen` with a new function:

```ts
isModernRampCountry(countryCode)
```

It should return true for Koywe countries plus `US`.

Do not hard-route US users to `LegacyGuardarianTopUpScreen` after Coinbase is enabled. US should use the modern method picker and include both Coinbase ACH and Guardarian card/mobile-pay options.

### 2. Availability UI

For US, show two methods:

- Recommended: Coinbase / ACH
- Fallback: Debit card / Apple Pay / Google Pay via Guardarian

Copy should be English or Spanish-neutral depending on current app localization. Since this screen is Spanish today, V1 can use Spanish copy:

- "Coinbase / ACH"
- "Tarjeta, Apple Pay o Google Pay"

The UI should not promise "0.5%" as static copy. It can say "menor comisión estimada" or show the provider quote. Coinbase and Guardarian own final fees.

### 3. Order Handling

Reuse `CREATE_RAMP_ORDER`.

Branch by response:

- If `nextActionUrl`: open hosted Coinbase flow or navigate to an in-app browser screen if that is the existing pattern.
- For Guardarian hosted card/mobile-pay, reuse the existing hosted redirect behavior but feed it through the modern ramp selector and `RampTransaction` history path.

### 4. Completion UX

Reuse `RampInstructionsScreen` for hosted or pending-status views.

```
Provider hosted checkout opened
  |
  v
Waiting for USDC on Algorand after return/webhook
  |
  v
Balance refresh / conversion status
```

Do not promise instant cUSD. The real invariant is "Coinbase sends USDC to your Confio wallet"; cUSD availability depends on existing autoswap/conversion.

## NOT In Scope

- Generic provider registry across Koywe, Coinbase, and Guardarian. Useful later, too much for V1.
- Coinbase off-ramp UI. Foundation can support `provider=coinbase`, but user-facing sell flow waits until onramp is stable.
- Replacing Guardarian globally. Guardarian remains the US card/mobile-pay fallback and fallback outside supported Koywe/Coinbase routes.
- Rewriting Koywe. Touch only shared surfaces needed for Coinbase.
- New custody model. Coinbase always settles to the user's non-custodial Algorand address.
- Headless Onramp. Avoids Coinbase's 60-day US phone re-verification requirement, native payment UI, and Apple/Google Pay setup in V1.

## Failure Modes

| Failure | Handling | Test |
| --- | --- | --- |
| Coinbase credentials missing | Availability hides Coinbase or order returns clear error | Unit |
| Coinbase buy options do not include USDC Algorand | Availability disables Coinbase and logs config issue | Unit |
| Guardarian card checkout fails for US | User sees retryable provider error and can choose Coinbase ACH | Unit/integration |
| Active account has no Algorand address | `createRampOrder` fails before provider call | Unit |
| User switches active account between quote and order | Order uses JWT active account at creation time, not client address | Unit |
| Coinbase create-order timeout | No `RampTransaction` unless provider order ID exists; user sees retryable error | Unit |
| Coinbase returns order but DB write fails | Log exception; no false success to client | Unit |
| Duplicate webhook | `RampWebhookEvent.get_or_create` returns duplicate OK | Unit |
| Unknown webhook order | Store event, return OK, alert/log for reconciliation | Unit |
| Status regresses from COMPLETED to PROCESSING | Status order prevents downgrade | Unit |
| User closes hosted Coinbase checkout | Webhook/status polling updates transaction; user can resume from pending ramp state | Integration |
| Coinbase completes USDC but autoswap fails | Existing USDC balance remains visible; conversion retry path handles cUSD | Integration |

## Test Plan

### Backend Unit Tests

Add `ramps/tests/test_coinbase_client.py`:

- Builds hosted order payload with `destinationAddress`, `destinationNetwork=algorand`, `asset=USDC`.
- Parses Coinbase quote/order response.
- Converts Coinbase API errors to safe app errors.
- Refuses unsupported asset/network option.

Add `ramps/tests/test_coinbase_sync.py`:

- Signature verification success/failure.
- Event ID extraction.
- Order ID extraction.
- Status mapping for pending, processing, completed, failed, review.
- Completed status cannot downgrade.
- Duplicate webhook is safe.
- Unknown order stores event and returns OK.

Add `ramps/tests/test_coinbase_schema.py`:

- `rampAvailability(countryCode: "US")` returns Coinbase ACH and Guardarian card/mobile-pay methods.
- US availability is hidden when `COINBASE_ONRAMP_ENABLED=false`.
- `createRampOrder` for `COINBASE_HOSTED_ACH` stores `RampTransaction(provider=coinbase)`.
- `createRampOrder` for `GUARDARIAN_HOSTED_CARD` stores or links `RampTransaction(provider=guardarian)`.
- `createRampOrder` sets the destination from server-side account, not client input.
- Missing Algorand address returns a clear error.
- Unsupported direction for V1 off-ramp returns a clear error.
- Order status only returns transactions owned by the active user/account.

### Backend Integration Tests

Add webhook tests around `POST /api/coinbase/onramp/webhook/`:

- Valid completed webhook updates `RampTransaction` to `COMPLETED`.
- Duplicate webhook returns OK and does not reprocess.
- Invalid signature returns 403.
- Unknown order returns OK and stores `RampWebhookEvent`.

### Mobile Tests

Add or extend tests around:

- `isModernRampCountry("US")` is true.
- `TopUpScreen` uses modern ramp path for US.
- US methods render Coinbase ACH and Guardarian card/mobile-pay labels.
- Hosted Coinbase response opens/navigates to hosted flow.
- Guardarian hosted response opens/navigates to hosted flow.
- Error states show retryable copy for provider failures.

## Performance Review

No high-volume path risk if implemented as above.

Rules:

- Cache Coinbase buy options for 5-10 minutes.
- Do not call Coinbase options endpoint on every render if US availability is requested repeatedly.
- Webhook handler should do one event insert, one transaction lookup, and one transaction update.
- Status polling should be user-triggered or limited, not a tight client interval.

## Worktree Parallelization Strategy

Parallelizable after the model/settings migration is agreed.

| Step | Modules touched | Depends on |
| --- | --- | --- |
| Backend Coinbase client/sync | `ramps/` | model/provider choices |
| Webhook endpoint | `ramps/`, `config/` | sync helpers |
| GraphQL availability/order/status | `ramps/`, `config/` | client/sync helpers |
| Mobile routing/UI | `apps/src/` | GraphQL response shape |
| Tests | `ramps/tests/`, `apps/src/` | each owning module |

Lanes:

- Lane A: backend model/settings -> Coinbase client/sync -> GraphQL.
- Lane B: mobile routing/UI can start after GraphQL response shape is stubbed.
- Lane C: webhook tests and schema tests follow Lane A.

Execution order:

1. Land backend response contract first.
2. Build mobile against that contract.
3. Finish webhook/status reconciliation.
4. Run end-to-end sandbox order through Coinbase.

Conflict flags:

- Backend work should stay sequential inside `ramps/` because `schema.py` is already large and merge conflicts are likely.
- Mobile can run separately once the GraphQL contract is fixed.

## Open Decisions

1. Should US users skip Confio `RampAddress` requirements entirely for Coinbase, relying on Coinbase KYC/payment checks?
2. Should Coinbase off-ramp be visible as "coming soon" or absent until the first onramp cohort works?
3. Should Guardarian US card/mobile-pay be shown as a separate card in the same screen, or nested under "Otros métodos" to keep Coinbase ACH as the obvious default?

## Recommendation

Build V1 as:

- Hosted Coinbase ACH onramp enabled for US and shown as the recommended/default method.
- Guardarian hosted card/mobile-pay kept as a US fallback method in the same modern screen.
- Coinbase off-ramp absent from UI.
- All orders settled directly to the active account's server-resolved Algorand address.
- Full backend tests before production access.

This is the smallest complete version: it avoids making Guardarian's higher-fee card path the default for US users, gives ACH users the Coinbase path, preserves a card/mobile-pay fallback, preserves non-custodial settlement, and avoids the 60-day phone OTP burden of Headless Onramp.
